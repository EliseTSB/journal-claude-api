Journal Claude API

A real-time conversational AI service accessible over WebRTC data channels. This documentation describes how to integrate it into your application.

Version v1  ·  Real-time streaming  ·  WebRTC / PeerJS transport

Overview

The Journal Claude API provides conversational AI access through a peer-to-peer WebRTC connection. Clients establish a data channel to the service, exchange JSON-formatted messages, and receive streamed responses.

Key characteristics

  • Real-time streaming. Responses arrive as they are generated, chunk by chunk, over the data channel.
  • Session continuity. Each authenticated client maintains conversation context across multiple prompts within a session.
  • Unreliable transport. Data channels are configured for low-latency unreliable delivery. Clients implementing more than fire-and-forget interactions must build reliability themselves (see Reliability & Sequencing).
  • PeerJS-compatible. Clients may use the PeerJS library for connection management, or implement raw WebRTC directly.

This service is not appropriate for

  • High-availability production applications (no SLA)
  • Bulk content generation or scraping
  • Use cases requiring strict ordered delivery without client-side reassembly logic
Before you start

Your service peer ID, username, and password are delivered to you directly by the operator (in your welcome email). The examples below use your-service-peer-id as a placeholder — substitute the real value from your welcome email.

Quickstart

The minimal client connects to the service, logs in, sends a prompt, and prints the streamed response.

// Minimal client using PeerJS. `sign()` + your keypair come from the Authentication section.
const SERVICE_PEER_ID = 'your-service-peer-id';   // from your welcome email
const peer = new Peer();

peer.on('open', () => {
  // serialization: 'raw' sends each message as a JSON string, as-is (required).
  const conn = peer.connect(SERVICE_PEER_ID, { reliable: false, serialization: 'raw' });
  let outSeq = 0;

  // Every outbound message is signed; post-login messages carry the session_token.
  async function send(msg) {
    msg.seq = outSeq++;
    msg.sig = await sign(privateKey, msg);
    conn.send(JSON.stringify(msg));
  }

  conn.on('open', () => {
    send({ type: 'login', username: 'your-username', password: getPasswordAtRuntime() });
  });

  conn.on('data', (raw) => {
    const msg = JSON.parse(raw);
    if (msg.type === 'login_ok') {
      send({ type: 'prompt', session_token: msg.session_token,
             content: 'Hello! What can you help me build?' });
    } else if (msg.type === 'response_chunk') {
      process.stdout.write(msg.content);
    } else if (msg.type === 'response_end') {
      console.log('\n[response complete]');
    } else if (msg.type === 'login_failed' || msg.type === 'error') {
      console.error('Error:', msg.code, msg.message);
    }
  });
});
Heads up

This example assumes well-ordered, lossless delivery — which the service does not guarantee. For anything beyond a first experiment, see Reliability & Sequencing.

Authentication

Authentication is a three-step flow. Once the data channel opens, you (1) sign every message with your signing key, (2) log in with your username and password to obtain a short-lived session token, and (3) include that session token on every subsequent request.

Message signing

Each client owns a signing keypair. You generate it once, keep the private key, and register your public key with the service (contact your administrator). The service only ever holds your public key.

Every message you send must carry a sig field — an ECDSA (P-256, SHA-256) signature over a canonical serialization of the message, made with your private key. The canonical signing input is the newline-joined string:

<type>\n<seq>\n<session_token or "">\n<SHA-256 hex of content, or "">

sig is the resulting signature, base64url-encoded. The service rebuilds the same string, verifies the signature against your registered public key, and silently drops any message with a missing or invalid signature before any other processing.

Generate and register your keypair once:

const kp = await crypto.subtle.generateKey(
  { name: 'ECDSA', namedCurve: 'P-256' },
  false,                          // private key non-extractable
  ['sign', 'verify']
);
const publicJwk = await crypto.subtle.exportKey('jwk', kp.publicKey);
// Register publicJwk with your administrator. Persist the keypair (e.g. IndexedDB)
// so it survives reloads. Keep the private key private.

Sign each message:

async function sign(privateKey, msg) {
  const enc = new TextEncoder();
  const contentHash = msg.content
    ? [...new Uint8Array(await crypto.subtle.digest('SHA-256', enc.encode(msg.content)))]
        .map(b => b.toString(16).padStart(2, '0')).join('')
    : '';
  const input = `${msg.type}\n${msg.seq}\n${msg.session_token || ''}\n${contentHash}`;
  const buf = await crypto.subtle.sign({ name: 'ECDSA', hash: 'SHA-256' }, privateKey, enc.encode(input));
  return btoa(String.fromCharCode(...new Uint8Array(buf)))
    .replace(/\+/g, '-').replace(/\//g, '_').replace(/=+$/, '');   // base64url
}
Security

Message signing protects the service from unauthenticated traffic. It identifies your application (the holder of the private key), not you — that's what login is for. Keep your private key in client-side storage only; never commit or log it.

Login

After the channel opens, send a login message:

{
  "type": "login",
  "seq": 0,
  "username": "your-username",
  "password": "your-password",
  "sig": "<signature over this message>"
}

The service responds with one of:

  • login_ok — credentials accepted; carries your session_token, session_id, rate_limit, and max_packet_size
  • login_failed — invalid credentials. Repeated failures trigger a temporary lockout (see Error Codes).
Security

Enter your password at runtime (e.g., a login prompt in your application) rather than storing it in your source. A password embedded in client-side code is not secret.

Session token

login_ok returns a session_token:

{
  "type": "login_ok",
  "seq": 0,
  "session_id": "sess_abc123",
  "session_token": "st_live_9f3a8b2c1d",
  "rate_limit": {
    "requests_per_hour": 120,
    "tokens_per_day": 100000
  },
  "max_packet_size": 16384
}

Include the session_token on every subsequent message (prompt, history_request, etc.), alongside the per-message sig:

{
  "type": "prompt",
  "seq": 5,
  "session_token": "st_live_9f3a8b2c1d",
  "content": "Explain what a closure is in JavaScript.",
  "sig": "<signature over this message>"
}

Session tokens are short-lived. When a token expires, log in again to obtain a new one. To resume a session after disconnection, see Reconnection.

Credential storage

Your signing key and session token are sensitive. Recommended practices:

  • Do not commit credentials to version control
  • Do not embed your password in source — enter it at runtime
  • Do not log credentials in client output
  • Treat the session token like a password for its lifetime

Connection

Service discovery

The service is reachable on the PeerJS network at the peer ID provided in your welcome email. Connect to it as:

const SERVICE_PEER_ID = 'your-service-peer-id';   // from your welcome email
const conn = peer.connect(SERVICE_PEER_ID, { reliable: false, serialization: 'raw' });

Implementing raw RTCPeerConnection instead of PeerJS? Contact the operator for signaling details.

Data channel configuration

The data channel is negotiated with the following configuration:

ParameterValue
reliablefalse  (→ ordered: false, maxRetransmits: 0)
serialization'raw'  (messages are sent as JSON strings, as-is)
serialization: 'raw' is required

The service exchanges newline-delimited JSON strings. PeerJS defaults to a binary serializer that wraps your payload — which the service will not understand. Always pass serialization: 'raw' in your peer.connect() options so messages travel as plain strings.

Implications for clients

  • Messages may arrive out of order
  • Messages may be dropped without notification
  • The client is responsible for sequencing, retransmission requests, and reassembly

Connection lifecycle

[1]  Client opens PeerJS peer
[2]  Client.connect(serviceId, { reliable: false, serialization: 'raw' })
[3]  Service.on('connection') fires
[4]  Data channel opens
[5]  Client sends signed { type: 'login', username, password }
[6]  Service responds { type: 'login_ok', session_id, session_token }
[7]  --- Session active ---
[8]  Client sends prompts; service streams responses
[9]  Either party may send { type: 'close', reason: '...' }
[10] Data channel closes

Heartbeats

The service sends a ping message every 30 seconds when the channel is idle:

{ "type": "ping", "ts": 1716406523456 }

Clients should respond with pong:

{ "type": "pong", "ts": 1716406523456 }

If the service does not receive a pong within 60 seconds of a ping, it considers the connection dead and closes the data channel.

Reconnection

After unexpected disconnection, clients may reconnect within 5 minutes and resume the previous session. Session state (conversation context) persists across reconnections within this window.

To reconnect:

  1. Re-establish the data channel as in initial connection
  2. Send a login message (signed, as always)
  3. Include the previous session_id in resume_session to request resume:
{
  "type": "login",
  "seq": 0,
  "username": "your-username",
  "password": "your-password",
  "resume_session": "sess_abc123",
  "sig": "<signature>"
}

On successful resume:

{
  "type": "login_ok",
  "seq": 0,
  "session_id": "sess_abc123",
  "session_token": "st_live_9f3a8b2c1d",
  "resumed": true
}

If the session has expired or cannot be resumed, the service issues a fresh session_id (and a new session_token).

Reliability & Sequencing

What the channel guarantees

  • Delivery: packets may be lost without notification
  • Ordering: packets may arrive in any order
  • Duplication: packets may arrive more than once (rare but possible)
  • Integrity: packets that do arrive are not corrupted (DTLS provides this)

What clients must implement

If your application requires more than fire-and-forget prompts, you must implement:

  1. Sequence tracking. Track the highest contiguous seq received from the service. Detect gaps.
  2. Acknowledgment. Periodically send ack messages with the highest contiguous sequence received.
  3. Retransmission requests. When gaps are detected, send retransmit_request for the missing indices.
  4. Chunk reassembly. Buffer response_chunk messages by response_id; reassemble in order by chunk_index when all chunks arrive.
  5. Duplicate detection. Reject messages with sequence numbers you have already processed.
  6. Timeout handling. If a response is incomplete after a reasonable interval, decide whether to request retransmits or give up.
  7. Backpressure. Throttle your prompt rate based on observed response latency. The service applies its own backpressure on outbound responses (see below).

Backpressure semantics

The service tracks per-connection how many response_chunk messages it has sent without receiving an ack covering them.

  • Window size: 32 unacked chunks
  • When the window fills: the service pauses sending further response_chunk messages for this response_id
  • When the client sends an ack: the service resumes sending if the unacked count drops below the window

Clients implementing reliable transport SHOULD send ack messages with reasonable frequency (e.g., on every Nth chunk received, or after a short timer) to prevent the service from pausing mid-response. Clients that never send ack messages will receive at most 32 chunks per response before the service pauses indefinitely.

For long responses (>32 chunks), failing to ack will appear as a hung response. The service does not time out the response itself, but does require client cooperation to continue streaming.

Suggested implementation pattern

A clean implementation typically maintains:

  • An inbound buffer keyed by (response_id, chunk_index)
  • An outbound buffer of unacknowledged messages (your own seq numbers)
  • A next-expected-sequence counter for the service direction
  • Per-response state: total chunks expected, chunks received, last activity timestamp
  • A timer triggering retransmission requests after seq gaps persist past a threshold

The service's retransmit buffer is finite — request missing chunks promptly (within 30 seconds) or they will no longer be available.

History & Retrieval

The service stores the full history of your conversations. You can retrieve it on demand — to display past messages, or to download your entire history for local processing (search, export, analysis).

History is delivered over the same unreliable data channel as live responses, using the same chunking and sequencing. Your reliability layer (sequence tracking, reassembly, retransmit) applies to history exactly as it applies to responses — a history page is just another chunked, streamed payload. You may request history at any time, including while a live response is streaming — history retrieval runs independently of the live conversation flow and does not count against the concurrent-response limit.

Requesting history

{
  "type": "history_request",
  "seq": 12,
  "cursor": null,
  "page_size": 25
}
  • cursor (string or null): the pagination cursor. Pass null for the most recent page. To reach older messages, pass the next_cursor from the previous history_end (below).
  • page_size (number, optional): requested number of message-pairs per page. The service may return fewer; it will never return more than its maximum. Default 25.

History is paginated backward in time: the first page (cursor: null) is your most recent activity; each subsequent page reaches further into the past.

Receiving a history page

A page is streamed as one or more history_chunk messages, followed by a single history_end:

{
  "type": "history_chunk",
  "seq": 40,
  "page_id": "hist_pg_7f3a",
  "chunk_index": 0,
  "content": "<a slice of the page's serialized messages>"
}
  • page_id (string): identifies this page. Associate chunks by (page_id, chunk_index), exactly as you do with response_id for responses.
  • chunk_index (number): ordering within the page, starting at 0.
  • content (string): a slice of the page payload. Reassemble all chunks in chunk_index order to get the full page.

The reassembled page payload is a JSON array of message records, oldest-first within the page:

[
  {
    "ts": "2026-05-31T22:14:18Z",
    "channel": "rtc",
    "role": "user",
    "content": "How do I implement a retransmit buffer?"
  },
  {
    "ts": "2026-05-31T22:14:31Z",
    "channel": "rtc",
    "role": "assistant",
    "content": "Start by asking what the receiver needs to track..."
  }
]

The page ends with:

{
  "type": "history_end",
  "seq": 52,
  "page_id": "hist_pg_7f3a",
  "chunk_total": 12,
  "next_cursor": "eyJiZWZvcmUiOiAxNzE2..."
}
  • chunk_total (number): how many history_chunk messages this page sent. You should have received chunks 0 through chunk_total - 1; request retransmits for any gaps before assembling.
  • next_cursor (string or null): pass this as cursor in your next history_request to get the next-older page. When next_cursor is null, you have reached the beginning of your history — there is nothing older.

Cursor semantics

The cursor is an opaque token. Do not parse it, construct it, or depend on its internal format — it may change without notice. Treat it as a black box: the service hands you a next_cursor, you hand it back verbatim. The only special value is null (as a request cursor: start at the most recent page; as a received next_cursor: end of history).

History is immutable and stable for backward paging: paging into the past returns the same messages in the same order every time. New messages you send or receive arrive through the live response_* flow, not through history — so a backward page-walk is never disturbed by new activity.

Downloading your full history

To obtain your complete history (for example, to build local search), page backward until next_cursor is null:

  1. history_request with cursor: null.
  2. Reassemble the page; append its records to your local store.
  3. If next_cursor is not null, send history_request with cursor: next_cursor; repeat from step 2.
  4. Stop when history_end.next_cursor is null — you now hold every message, exactly once.
Guarantee

A complete backward walk yields every message exactly once — no gaps, no duplicates.

Rate Limits

These limits exist to protect the service from runaway clients — an infinite loop, a missing backpressure check, a request storm — not to ration your usage. They are set generously: a frontend built with reasonable care will never reach them. If you do hit one during normal use, it almost always means a bug in your client (a loop, a missing throttle), not that you're using the service "too much."

The service enforces the following per token:

LimitDefaultNotes
Requests per hour120A "request" is one prompt (chunked prompts count as one). history_requests do not count.
Output tokens per day100,000Sum of output_tokens across all responses
Concurrent active responses1One prompt response in-flight at a time. History fetches are exempt — a history_request may run while a response is streaming.
Maximum prompt length100,000 charsAcross all chunks of a chunked prompt
Maximum session duration24 hoursSessions reset after 24 hours of activity

When a limit is hit, the service returns a rate_limited message with retry_after indicating when the request may be retried. During normal use, treat any rate_limited as a prompt to look for a loop or a missing throttle in your own code.

Rate-limit values are surfaced in the login_ok message at session start.

Code Examples

JavaScript (PeerJS) — client with reliability scaffolding

const SERVICE_PEER_ID = 'your-service-peer-id';   // from your welcome email

class JournalClaudeClient {
  constructor({ username, password, privateKey }) {
    this.username = username;
    this.password = password;       // enter at runtime; don't hardcode
    this.privateKey = privateKey;   // your ECDSA private key (see Authentication)
    this.peer = new Peer();
    this.conn = null;
    this.session_id = null;
    this.session_token = null;
    this.outbound_seq = 0;
    this.next_expected_seq = 1;
    this.responses = new Map(); // response_id -> { chunks: Map, total: null, callbacks }
    this.pending_acks = new Map(); // seq -> message
  }

  async connect() {
    return new Promise((resolve, reject) => {
      this.peer.on('open', () => {
        this.conn = this.peer.connect(SERVICE_PEER_ID, { reliable: false, serialization: 'raw' });

        this.conn.on('open', () => {
          // First message: signed login. The session_token comes back in login_ok.
          this.send({ type: 'login', username: this.username, password: this.password });
        });

        this.conn.on('data', (raw) => {
          try {
            const msg = JSON.parse(raw);
            this.handleMessage(msg, resolve, reject);
          } catch (e) {
            console.error('Malformed message:', e);
          }
        });

        this.conn.on('error', reject);
      });
    });
  }

  // Every outbound message is signed; post-login messages also carry the session_token.
  async send(msg) {
    msg.seq = this.outbound_seq++;
    if (this.session_token && msg.type !== 'login') msg.session_token = this.session_token;
    msg.sig = await sign(this.privateKey, msg);   // sign() from the Authentication section
    this.conn.send(JSON.stringify(msg));
    return msg.seq;
  }

  handleMessage(msg, resolveLogin, rejectLogin) {
    switch (msg.type) {
      case 'login_ok':
        this.session_id = msg.session_id;
        this.session_token = msg.session_token;
        resolveLogin(msg);
        break;
      case 'login_failed':
        rejectLogin(new Error(msg.message));
        break;
      case 'response_chunk':
        this.handleChunk(msg);
        break;
      case 'response_end':
        this.handleEnd(msg);
        break;
      case 'ping':
        this.send({ type: 'pong', ts: msg.ts });
        break;
      case 'error':
      case 'rate_limited':
        this.handleError(msg);
        break;
    }
  }

  handleChunk(msg) {
    let resp = this.responses.get(msg.response_id);
    if (!resp) {
      resp = { chunks: new Map(), total: null, onChunk: null, onEnd: null };
      this.responses.set(msg.response_id, resp);
    }
    resp.chunks.set(msg.chunk_index, msg.content);
    if (resp.onChunk) resp.onChunk(msg.content, msg.chunk_index);
  }

  handleEnd(msg) {
    const resp = this.responses.get(msg.response_id);
    if (!resp) return;
    resp.total = msg.chunk_total;
    if (resp.onEnd) {
      const ordered = [];
      for (let i = 0; i < resp.total; i++) {
        ordered.push(resp.chunks.get(i) || '[MISSING]');
      }
      resp.onEnd(ordered.join(''), msg.usage);
    }
  }
}
What this leaves for you

This example shows signing + login + session-token handling, but still omits ack/retransmit logic, sequence-gap detection, response_id correlation, reconnection, and backpressure. A complete implementation requires building these on top — see Reliability & Sequencing.

Minimum viable fire-and-forget client

// Simplest client. Assumes reliable delivery (which the service does NOT guarantee).
// `sign()` + your keypair come from the Authentication section.
const SERVICE_PEER_ID = 'your-service-peer-id';   // from your welcome email
const peer = new Peer();
peer.on('open', () => {
  const conn = peer.connect(SERVICE_PEER_ID, { reliable: false, serialization: 'raw' });
  let seq = 0;
  const send = async (m) => { m.seq = seq++; m.sig = await sign(privateKey, m); conn.send(JSON.stringify(m)); };
  conn.on('open', () => send({ type: 'login', username: 'you', password: getPassword() }));
  conn.on('data', (raw) => {
    const m = JSON.parse(raw);
    if (m.type === 'login_ok') send({ type: 'prompt', session_token: m.session_token, content: 'Hi!' });
    else if (m.type === 'response_chunk') console.log(m.content);
  });
});
